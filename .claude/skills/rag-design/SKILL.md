# RAG Design Skill

RAGシステムの設計・構築を包括的に支援するskill。  
前処理（Data Parsing & Indexing）と検索（Retrieval）の2フェーズをカバーする。  
生成フェーズの改善については `rag-evaluation` skillを参照。

---

## RAGの3分類

| 分類 | 概要 | 用途 |
|---|---|---|
| **Naive RAG** | ユーザ質問 → ベクトル検索 → プロンプト付与 → LLM回答 | PoC・シンプルな用途 |
| **Advanced RAG** | Pre/Post-Retrievalの処理を追加し精度向上 | 本番システム |
| **Modular RAG** | 各機能をモジュールとして組み合わせ | 複雑な要件 |

## フェーズ1：前処理（Data Parsing & Indexing）

### Data Parsing

**OCR・構造化出力**
- 正規化処理で取り除くべきノイズ：重複空白、ページ番号、ヘッダ/フッタ、改行崩れ、ハイフネーション、コピー防止用不可視記号
- 保存フォーマットの選択：
  - **JSON**：スキーマの意味合いを適切に伝えられる
  - **HTML**：表の関係性を明示できる（表形式データはRAGが苦手なため特に有効）
  - 表形式データの詳細：[arxiv.org/abs/2402.17944](https://arxiv.org/abs/2402.17944)

### チャンキング戦略

**チャンクサイズの選び方**

| サイズ | 問題 |
|---|---|
| 大きすぎる | 無駄な情報が流入し、recallが低下 |
| 小さすぎる | 文脈不足で回答が断片的になる。計算コスト増 |
| **512tokens（推奨）** | 最初の設定値として適切 |

**Semantic Chunking（意味単位での分割）**
- 固定長ではなく「意味の切れ目」で分割する手法
- 文ごとに埋め込みを取り、隣り合う文の類似度が落ちる点を境界とする
- デメリット：類似度算出とクラスタリングで計算コストが増大

### ベクトルDB選定

| DB | 特徴 | 推奨用途 |
|---|---|---|
| **Chroma** | 軽量 | PoC |
| **Qdrant** | 性能とスピードのバランスが良い | 実務全般 |
| **Faiss** | Meta開発、検索スピード最速 | 研究・大規模プロダクション |
| **LanceDB** | マルチモーダル（画像・表データ）に強い | 非テキストデータを含む場合 |

**設計原則**
- すべてを1つの大きなDBにまとめるより、**用途・内容ごとに複数DBに分けてエージェントにルーティングさせる**方が精度が高い

**インデックスフォーマット**

| フォーマット | 特徴 |
|---|---|
| IndexFlatIP | ベクトルをそのまま格納。小規模・高精度 |
| IVFFlat / HNSW | クラスタリング・近似探索。大規模向き。高速だが精度が若干落ちる |

## フェーズ2：検索（Retrieval）

詳細な手法比較は `references/retrieval-methods.md` を参照。

### 手法選択ガイド

```
質問の性質は？
├─ キーワード完全一致が重要 → Hybrid Search（Dense + Sparse）
├─ 意味的な検索が中心 → Dense（ベクトル検索）のみ
├─ 検索精度が不十分 → Rerank / LLM Rerank を追加
├─ ユーザ入力が短い・曖昧 → Query Rewrite（HyDE推奨）
├─ 取得チャンクに無関係な情報が混じる → ChunkRAG
├─ エンティティ間の関係性が重要 → GraphRAG / RAPL
└─ 細かいチャンクで文脈が失われる → Parent Page Retrieval
```

### 文書タイプ別推奨

| 文書タイプ | 推奨手法 |
|---|---|
| FAQ・マニュアルなど構造化文書 | Parent Page Retrieval |
| 専門用語・固有名詞が多い | Hybrid Search + Query Rewrite |
| エンティティ・関係性が重要 | GraphRAG / RAPL（低速注意） |
| 複数データソース（DB・Web等） | Agentic RAG |

## 設計チェックリスト

Claude がRAG設計を支援する際は以下を確認すること：

1. **データの性質**：テキスト/表/画像？構造化/非構造化？
2. **規模**：文書数・更新頻度
3. **精度要件**：precision重視 or recall重視？
4. **レイテンシ要件**：リアルタイム？バッチ？
5. **コスト制約**：LLM呼び出し回数の許容範囲
