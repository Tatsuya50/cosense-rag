# 検索手法詳細リファレンス

## Hybrid Search（ハイブリッド検索）

- Dense（ベクトル検索）とSparse（キーワード検索）を組み合わせる
- キーワード検索は**完全一致**に強く、ベクトル検索は**意味的類似**に強い
- ブレンド配分は対象文書の性質に応じてチューニングする
- 参考：[Microsoft Azure Hybrid Search](https://learn.microsoft.com/en-us/azure/search/hybrid-search-overview)
- **実装コスト**：中

---

## Rerank（再ランキング）

- 近似検索で候補を出した後、高精度検索で再スコアリングする2段階構成
- **実装コスト**：低〜中

## LLM Rerank

1. ベクトル検索で候補チャンクを取得
2. 質問と各チャンクをLLMに入力
3. LLMが関連性スコアを付与
4. スコアをベクトルスコアと組み合わせて最終順位を決定

- **実装コスト**：中（LLM呼び出しが増える）

---

## Query Rewrite（クエリ書き換え）

- ユーザの質問文をLLMで書き直し、検索しやすい形に変換する
- **HyDE（Hypothetical Document Embeddings）**が有名
  - 仮説：質問文より「回答文」の方が検索対象文書と類似度が高い
  - LLMに仮の回答を生成させ、それを検索クエリとして使う
  - 論文：[arxiv.org/abs/2212.10496](https://arxiv.org/abs/2212.10496)
- **実装コスト**：低（手軽に試せる）

---

## ChunkRAG

- 取得チャンクをそのまま渡さず、LLMで関連性をフィルタリングしてから使う

1. 文書を細かいチャンクに分割
2. チャンクをLLMで評価・スコアリング
3. 低スコアチャンクを除去
4. 高スコアチャンクのみをLLMに渡して回答生成

- 論文：[arxiv.org/abs/2410.19572](https://arxiv.org/abs/2410.19572)
- **実装コスト**：中（LLM評価ステップが追加）

---

## GraphRAG

- クエリのエンティティをグラフにマッピングし、ノード間の関係をたどって関連情報を探索
- 類似度ではなく**関係性**を活かす点が特徴
- **メリット**：複雑な関係性を持つ知識に対して高精度
- **デメリット**：処理速度が非常に遅い
- **実装コスト**：高

## RAPL（GraphRAGの改善版）

- グラフ上の最短経路だけでなく**複数経路**を探索して回答精度を向上
- GraphRAGより処理コストが高いが、的中率が上がる
- **実装コスト**：高

---

## Parent Page Retrieval（親チャンク取得）

- 細かいチャンクで検索しつつ、LLMには親チャンク（より広い文脈）を渡す

1. 文書をチャイルドチャンクとして作成・インデクシング
2. チャイルドチャンクをまとめた親チャンクを作成
3. チャイルドチャンクで類似度算出・検索
4. ヒットしたチャイルドの**親チャンク**を使って回答生成

- **有効な文書**：FAQ・マニュアルなど構造化された文書
- **効果が薄い文書**：一文が独立した短い資料（例：営業活動ログ）
- **実装コスト**：中

---

## Contextual Retrieval（コンテクスト付与）

- 各チャンクに「文書全体における役割」の説明を付与してから保存・検索する

1. 文書全体と対象チャンクをLLMに渡す
2. LLMが説明コンテクストを生成してチャンクに付与
3. コンテクスト付きチャンクを保存・検索に活用

- **メリット**：チャンク単体ではわからない文脈を補完
- **デメリット**：インデクシング時のLLM処理が非常に重い
- **実装コスト**：高
