# 生成フェーズ（Generation）改善手法詳細

## Self-RAG

検索の要否・根拠の品質・回答の整合性をLLM自身に判断させる手法。

### ステップ

**1. Gate（検索要否の判定）**
- クエリを受けたら「検索が必要か/不要か」を判定する
- 理由付けさせると閾値調整が楽になる
- 検索が必要なケースの例：固有名詞・日付・社内特有のワードを含む質問

**2. Retrieval（検索）**
- 必要と判断した場合、通常の検索ステップを実行

**3. Critique（自己批評）**
- 取得した根拠チャンクと下書き回答をLLMに自己批評させる
- 「根拠が十分か」「回答が矛盾していないか」を評価
- 不十分なら再検索・再生成を行う

**効果**：一貫性・説明可能性の向上  
**コスト**：中（LLM呼び出しが複数回）

## Agentic RAG

複数のデータソースを使い分けながら回答を生成するエージェント型構成。

### ステップ

1. **Agentに検索方法を判断させる**
   - Vector DBへの検索か、RDBへのSQLか、Web検索か、などを自律的に選択
2. **関連結果の評価**
   - 検索結果の妥当性をLLMに評価させる
   - 不足なら別ツール・別クエリで取り直す（反復的な検索）
3. **LLMへの入力**
4. **回答生成（＋最終チェック）**

### 特徴

- **複数データソースの使い分け**が最大の特徴
  - 例：定量データ（RDB/SQL）と定性データ（Vector DB）を分けてルーティング
  - 用途に応じてインターフェースを分けることで精度が安定する
- 広義には「ローカルフォルダ検索」「Web検索」もRAGのツールとして組み込める

**効果**：複雑なクエリへの対応力・多様なデータソース活用  
**コスト**：高（エージェント設計が必要）

## RAG Reasoning（発展途上）

RAGと推論（Reasoning）を組み合わせた発展的手法。現時点では研究段階のものも含む。

### 3つのアプローチ

**1. Reasoning-Enhanced RAG**
- まずLLMに「回答に必要な情報は何か？」を推論させる
- その内容をもとに複数の検索クエリを自動生成
- 候補クエリを評価し、適切なものを使って検索
- 検索の質がLLMの推論能力に依存する

**2. RAG-Enhanced Reasoning**
- RAGで取得した情報を「推論の材料」として直接使う
- 単なる回答生成ではなく、推論プロセス自体にRAGを組み込む

**3. Synergized RAG-Reasoning**
- 上記2つを組み合わせた統合アプローチ
- 最も高度だが、実装・評価が複雑

**効果**：高度な多段階推論を要するクエリへの対応  
**コスト**：高（研究的・実験的な段階）
